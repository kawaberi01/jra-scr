# 設計判断

## 1. `cname` は外部 API に公開しない

- JRA 固有の内部導線知識を外へ漏らさない
- 公開 API は `date`、`course`、`race_no`、`race_id` を主に使う

## 2. JRA 固有の遷移知識を `navigation` に分離する

- `provider` と `service` に散らさない
- サイト変更時の修正範囲を狭くする

## 3. オッズ取得は `race × bet_type` を基本単位にする

- 全券種一括取得は重い
- JRA の導線も券種単位に分かれる
- cache key として扱いやすい

## 4. fixture first で実装し、live fetch は後追いで安定化する

- まず regression を固定したい
- スクレイピング基盤は fixture がないと保守しにくい

## 5. 結果・払戻は `accessH` の開催単位ページを正とする

- `accessS` 単独では払戻が不足した
- `accessH` 側から結果と払戻を同時抽出する方が安定した

## 6. 過去結果バッチは骨組みを先に入れる

- 収集導線とユースケースを先に固定する
- 保存先や retry は後から差し込む
