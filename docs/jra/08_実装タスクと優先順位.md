# JRA API 実装タスクと優先順位

最終更新: 2026-06-28

## 優先順位 A

1. API 例外の HTTP レスポンス化
- 目的: `LookupError` や取得失敗を `500` のまま返さず、利用者が原因を判断できるようにする
- 完了条件:
  - 入力不正を `400`
  - 対象未発見を `404`
  - JRA / 外部取得失敗を `502`
  - API テスト追加

2. HttpProvider の retry / backoff / timeout 整備
- 目的: JRA 側の一時失敗に対して API を壊れにくくする
- 完了条件:
  - GET / POST 共通の retry 方針
  - timeout 超過時の扱い統一
  - テスト追加

3. 文字コードと文字化けの整理
- 目的: README、OpenAPI、JRA 取得本文の可読性を回復する
- 完了条件:
  - ソースと docs の UTF-8 統一
  - 文字化け箇所の修正

## 優先順位 B

4. batch 実行用 CLI 追加
- 目的: 過去結果収集を手作業なしで実行できるようにする
- 完了条件:
  - 日付範囲、開催地、出力先を CLI 指定
  - JSONL 保存と retry を利用

5. 蓄積済み結果の読み出し API 追加
- 目的: JSONL に保存した結果を検索・再利用できるようにする
- 完了条件:
  - `race_id` 指定取得
  - 日付範囲取得
  - 開催地絞り込み

6. ログ整備
- 目的: `course`、`race_id`、`cname` 単位で障害点を追えるようにする
- 完了条件:
  - API リクエストログ
  - provider 取得ログ
  - 失敗時の構造化ログ

## 優先順位 C

7. 永続キャッシュ
- 目的: 再起動後も再取得コストを抑える

8. ヘルスチェック強化
- 目的: プロセス生存だけでなく upstream 到達性も確認する

9. Enum ベースの入力バリデーション
- 目的: `course` と `bet_type` の OpenAPI と実装を揃える
