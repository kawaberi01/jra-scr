# アーキテクチャ

## 技術スタック

- Python 3.12
- FastAPI
- httpx
- BeautifulSoup4
- pytest
- fixture ベースの `FixtureProvider`

## 層構造

### `provider`

責務:

- HTML 取得
- `JRADB/access*.html` に対する GET / POST の再現
- fixture と live fetch の切り替え

主ファイル:

- `src/jra_srb/provider.py`

### `navigation`

責務:

- 開催選択ページから `course + date + kind` に対応する `cname` を解決する
- JRA 固有の `doAction(..., cname)` 知識を隔離する

主ファイル:

- `src/jra_srb/navigation.py`

### `extractors`

責務:

- HTML から meeting、出馬表、オッズ、結果・払戻を抽出する
- fixture 向け parser と JRA 実ページ向け parser を一箇所に持つ

主ファイル:

- `src/jra_srb/extractors.py`

### `service`

責務:

- ユースケース単位の処理
- JRA navigation と parser の接続
- 短 TTL キャッシュ
- 開催地座標から race_id を引き直す処理

主ファイル:

- `src/jra_srb/service.py`

### `delivery`

責務:

- FastAPI の公開入口
- JRA 固有知識を持たず、service を呼び出す

主ファイル:

- `src/jra_srb/app.py`

### `batch`

責務:

- 過去結果収集の最小ループ
- 将来の保存先 abstraction への接続点

主ファイル:

- `src/jra_srb/batch.py`

## フォルダ構成

- `src/jra_srb/`
  - 本体コード
- `src/jra_srb/config/parsers/`
  - parser 設定 JSON
- `tests/`
  - service / api / navigation / batch テスト
- `tests/fixtures/`
  - JRA fixture
- `docs/superpowers/`
  - design / plan / progress
- `docs/jra/`
  - formal docs

## データフロー

1. API が `date + course` または `race_id` を受ける
2. `service` が必要なら `navigation` を使って `cname` を解決する
3. `provider` が JRA ページまたは fixture を取得する
4. `extractors` が HTML を構造化する
5. `service` が Pydantic モデルへまとめる
6. FastAPI が返却する

## キャッシュ方針

- meeting
  - 短 TTL
- card
  - 中 TTL
- odds
  - `race_id + bet_type` 単位で短 TTL
- result
  - 長め TTL
