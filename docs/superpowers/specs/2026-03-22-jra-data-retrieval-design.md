# JRA Data Retrieval Design

## 目的

JRA 公式サイトから次の情報を安定取得できるローカル API 基盤を構築する。

- レース一覧
- 出馬表
- オッズ
- 結果・払戻

最終的には MCP ツールから自然言語で要求された内容を解釈し、この API を通じて確実に情報取得できる状態を目指す。

## 対象範囲

初期対象は以下の 2 系統とする。

1. オンライン取得
   - 当日または今週開催のレース一覧
   - 出馬表
   - 最新オッズ
   - 結果・払戻
2. 過去データ取得
   - 過去レースの結果・払戻を中心に、緩やかなバッチで収集

初期段階では過去オッズ履歴の保存は対象外とする。

## JRA 公式サイト構造の確認結果

2026-03-22 時点で確認した範囲では、JRA のレース関連ページは単純な固定 URL 群ではなく、`JRADB/access*.html` 配下の導線と `cname` パラメータを使った遷移で構成されている。

主な入口は以下の通り。

- `accessD.html`: 出馬表
- `accessO.html`: オッズ
- `accessS.html`: レース結果
- `accessH.html`: 払戻金

これらは直接 `GET` するだけでは `パラメータエラー` になり、実際にはフォーム送信または `CNAME/cname` 付きの遷移が必要である。開催選択画面には `2回中山8日` や `1回阪神10日` のような開催単位のリンクが並び、その先でレース単位へ進む構造になっている。

このため、JRA 固有の遷移ルールは専用のナビゲーション層に隔離する必要がある。

参考にした公式ページ:

- `https://jra.jp/`
- `https://jra.jp/JRAEN/AP/common/main`

## 採用方針

土台は「既存 GitHub 実装を参考にしつつ、`jra-srb` を中核に再構成する」とする。

理由は以下の通り。

- JRA 公式の導線は定型的で、既存実装の取得ノウハウを流用しやすい
- 最終目標が API 化と MCP 化であり、取得コードだけでなく公開インターフェース設計が必要
- オンライン取得と過去バッチ取得を同じ取得基盤で共有したい

つまり「既存コードの丸ごと採用」ではなく、「取得ロジックの流用 + API/MCP 向け境界の自前設計」を採る。

## アーキテクチャ

内部は次の 5 層に分ける。

### 1. navigation

JRA 固有の `JRADB + cname` 遷移を扱う。

責務:

- 開催選択ページの取得
- 開催地・開催日ごとの遷移先解決
- レース単位の遷移先解決

### 2. fetch

HTML を取得する。

初期実装:

- `httpx` ベース

将来拡張:

- 必要に応じて Playwright などを差し替え可能にする

### 3. parse

HTML を構造化データへ変換する。

対象:

- レース一覧
- 出馬表
- オッズ
- 結果・払戻

既存の JSON ベース設定駆動パーサを活用する。

### 4. service

ユースケース単位の処理をまとめる。

対象:

- 開催地単位の 1R-12R 一覧取得
- 単レース取得
- 最新オッズ取得
- 過去結果取得

### 5. delivery

外部公開層。

対象:

- FastAPI
- 将来の MCP tools

この層では JRA 固有知識を持たず、サービス呼び出しだけを行う。

## 入力パラメータ方針

外向き API は JRA 固有の `cname` を公開しない。

基本パラメータ:

- `date`: `YYYY-MM-DD`
- `course`: 英字コード
- `race_no`: `1` から `12`
- `race_id`: 内部で解決済みの場合に直接指定可能
- `bet_type`: オッズ券種
- `refresh`: 最新取得を優先するか
- `from_date` / `to_date`: 過去バッチ用

`course` は英字コードを正式仕様とする。

想定コード:

- `sapporo`
- `hakodate`
- `fukushima`
- `niigata`
- `tokyo`
- `nakayama`
- `chukyo`
- `kyoto`
- `hanshin`
- `kokura`

日本語表現の揺れは MCP または呼び出し側で正規化し、API 内部では正規コードだけを扱う。

## オンライン取得と過去取得の分離

### オンライン取得

目的:

- 当日または今週開催の情報を即時返却する

重視する点:

- 応答性
- 最新性

対象:

- レース一覧
- 出馬表
- 最新オッズ
- 結果・払戻

### 過去取得バッチ

目的:

- 過去レース結果・払戻を緩やかに収集し、将来活用可能な資産にする

重視する点:

- 欠損なく取得できること
- 開催単位・レース単位で再試行できること

対象:

- まずは結果・払戻
- 将来的には出馬表追加も可能

## 鮮度とキャッシュ方針

データ種別ごとに鮮度を分けて扱う。

### レース一覧

- 短い TTL を持つ
- 当日の開催状況変化には追従するが、秒単位更新は不要

### 出馬表・レース情報

- 長めの TTL を許容
- 確定後は準固定データとして扱う

### オッズ

- `race_id + bet_type` を基本取得単位とする
- 短い TTL のメモリキャッシュ
- `refresh=true` でキャッシュを無視して再取得可能にする

### 結果・払戻

- 未確定なら未公開として扱う
- 確定後は長め TTL

## オッズ取得方針

オッズの基本取得単位は「レース全体」ではなく「レース × 券種」とする。

理由:

- 3連複や3連単は全券種一括取得だと重い
- JRA 側の表示導線も券種単位で分かれている可能性が高い
- `race_id + bet_type` 単位のキャッシュが最も扱いやすい

推奨方式:

1. 指定レース・指定券種のオッズ一覧を一括取得
2. 正規化してメモリキャッシュに載せる
3. 必要なら組み合わせ条件で絞って返す

### オッズの返却モデル

券種ごとの正規化 JSON を返す。

例:

```json
{
  "race_id": "202603220611",
  "bet_type": "trifecta",
  "fetched_at": "2026-03-22T14:25:00Z",
  "source": "https://jra.jp/...",
  "entries": [
    {
      "combination": ["1", "2", "8"],
      "odds": "125.4",
      "popularity": "7"
    }
  ]
}
```

この構造により、将来的に以下の派生フィルタへ対応しやすくなる。

- 完全一致の組み合わせ
- ボックス条件
- 軸馬条件
- 人気順条件

## API の基本形

開催地単位と単レース単位を両方第一級の入口とする。

想定例:

- `GET /meetings/{date}/{course}`
- `GET /meetings/{date}/{course}/races/{race_no}/card`
- `GET /meetings/{date}/{course}/races/{race_no}/odds?bet_type=trifecta&refresh=true`
- `GET /meetings/{date}/{course}/races/{race_no}/result`
- `GET /races/{race_id}/card`
- `GET /races/{race_id}/odds?bet_type=trio&combination=1,4,7&refresh=true`

`bet_type` はオッズ API では必須寄りの設計とする。全券種一括返却は重くなりやすいため、必要なら将来追加する。

## エラー方針

最低限、以下を区別して返す。

- 該当開催なし
- 対象レース未確定
- 結果未公開
- JRA 構造変更による解析失敗
- 一時的取得失敗

MCP 利用時に「まだ公開されていない」のか「壊れている」のかを判別できることを重視する。

## 保存方針

初期段階では保存可能な境界だけ用意し、過剰な永続化は行わない。

- オンライン API:
  - メモリキャッシュ中心
- 過去取得バッチ:
  - ローカル保存を前提
- オッズ履歴:
  - 今回の対象外
  - 将来必要なら別バッチとして追加

この方針により、MVP を軽く保ちつつ、後から DB へ移行しやすい構造を維持する。

## 実装優先順位

1. JRA ナビゲーション層の確立
2. 開催地単位のレース一覧取得
3. 単レース出馬表取得
4. 券種単位オッズ取得
5. 結果・払戻取得
6. FastAPI 入口整備
7. 過去結果バッチ取得
8. MCP 連携

## 今回の設計で意図的に後回しにした項目

- 過去オッズ履歴保存
- 大規模クローリング基盤
- 分析用の平坦化スキーマ
- ブラウザ自動化の常用

これらは必要になった時点で追加する。
