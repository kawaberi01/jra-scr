# 成果物戦略と開発フロー

## 目的

この文書は、`jra-srb` において「最初に何を成果物として用意するか」と「それをどのように複数セッションで育てるか」を定義する。

このプロジェクトでは、1 セッションで設計から実装完了まで終わることを前提にしない。中断、別 PC、別セッション、別作業者での再開を前提とし、会話履歴ではなくリポジトリ上の成果物を正とする。

## 基本方針

- `docs/` は人が読む正式文書の置き場とする
- `docs/superpowers/` は AI 主導の作業文書の置き場とする
- 作業文書は中間成果物であり、最終成果物ではない
- 実装が進んだら、作業文書の内容を `docs/` 配下の正式文書へ反映する
- セッション継続は会話の記憶ではなく、ブランチ、差分、テスト、進捗メモ、正式文書で支える

## 文書の二層構造

### 1. 正式文書

`docs/` 直下に置く。固定ファイル名で管理し、プロジェクトの説明責務を持つ。

想定する主要文書:

- `docs/00_成果物戦略と開発フロー.md`
- `docs/01_プロジェクト概要.md`
- `docs/02_アーキテクチャ.md`
- `docs/03_設計判断.md`
- `docs/04_利用ガイド.md`
- `docs/05_コントリビューションガイド.md`
- `docs/06_セキュリティと運用注意.md`

必要に応じて追加してよいが、日付付きの乱立は避ける。

### 2. 作業文書

`docs/superpowers/` 配下に置く。テーマ単位、セッション単位の検討と実装を進めるための文書とする。

想定する主要文書:

- `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- `docs/superpowers/plans/YYYY-MM-DD-<topic>-implementation-plan.md`
- `docs/superpowers/YYYY-MM-DD-<topic>-progress-summary.md`

これらは作業を支えるために作るが、最終的な説明責務は `docs/` 側へ移す。

## 最初に決めるべき成果物

新しいテーマを始めるときは、最初に次を決める。

1. 今回のテーマ名
2. このテーマで新規作成または更新される正式文書
3. このテーマ専用の作業文書
4. セッションをまたいで残す進捗情報
5. 完了条件と検証手順

例: JRA スクレイピング基盤

- 正式文書:
  - `docs/02_アーキテクチャ.md`
  - `docs/03_設計判断.md`
  - `docs/04_利用ガイド.md`
  - `docs/06_セキュリティと運用注意.md`
- 作業文書:
  - 設計書
  - 実装計画
  - 進捗メモ

## Sessions の考え方

このプロジェクトでは、セッションとは「会話」ではなく「ある目的に紐づく一時的な作業実行」とみなす。

セッションが継続可能であるとは、次の状態が会話外に残っていることを意味する。

- どのブランチで作業しているか
- 必要ならどの worktree を使っているか
- 何を完了したか
- 何が未完了か
- 次の一手は何か
- どのテストが通っているか
- どの正式文書に未反映か

1 セッションごとに最低限残すべき情報:

- `Current status`
- `Completed`
- `Remaining`
- `Next step`
- `Verification`
- `Doc sync status`

この情報は `progress-summary` に集約する。

## Progress Summary の役割

`progress-summary` は単なるメモではなく、セッション継続のための最小ハブとする。

最低限含める内容:

- 目的
- 参照すべき設計書・実装計画
- 現在の状態
- 完了したこと
- 未完了
- 次の 1 手
- 実行した検証
- 注意点
- 正式文書への反映状況

セッション終了前には必ず更新する。

## Update-Docs の考え方

コードと作業文書だけを進めて、正式文書を後回しにしない。実装が進んだら、その変化を `docs/` 側へ定期的に反映する。

この反映をここでは `update-docs` と呼ぶ。

`update-docs` の目的:

- コードを source of truth としつつ、人が読める文書を遅れさせない
- `docs/superpowers` に閉じた知識を正式文書へ昇格させる
- 別セッションや別 PC で再開しても、`docs/` を読めば全体像が分かる状態を保つ

`update-docs` を行うタイミング:

- 大きな設計判断をしたとき
- 新しい API や操作方法が増えたとき
- モジュール構成が変わったとき
- セキュリティや運用上の注意が増えたとき
- セッション終了時に正式文書との乖離が大きいとき

## 標準フロー

### フェーズ 1. 成果物計画

最初に「何を作るか」ではなく「何を成果物として残すか」を決める。

決めること:

- 正式文書の更新対象
- 作業文書の作成対象
- 完了条件
- 検証方法

### フェーズ 2. 設計

`brainstorming` を使って個別テーマの設計を行う。

この段階で作るもの:

- テーマ設計書

この段階で同時に決めるもの:

- どの正式文書に反映されるべきか

### フェーズ 3. 計画

`writing-plans` を使って実装計画を作る。

この段階で作るもの:

- 実装計画
- 進捗メモの初版

### フェーズ 4. 実装

`test-driven-development`、`executing-plans`、`requesting-code-review` を使って進める。

この段階で更新するもの:

- コード
- テスト
- 進捗メモ

### フェーズ 5. 文書反映

節目ごとに `update-docs` を行う。

更新対象:

- アーキテクチャ
- 設計判断
- 利用ガイド
- 運用注意

### フェーズ 6. 完了処理

`finishing-a-development-branch` に入る前に、少なくとも次を満たす。

- 主要テストが通る
- `progress-summary` が最新
- 正式文書の未反映箇所が明記されている
- 次に読むべき文書が分かる

## セッション再開の標準手順

別 PC、別セッション、別作業者で再開するときは次の順で確認する。

1. `README.md`
2. `docs/00_成果物戦略と開発フロー.md`
3. 対応する正式文書
4. 対応する `design.md`
5. 対応する `implementation-plan.md`
6. 対応する `progress-summary.md`
7. `git status`
8. テスト実行

再開時に必要な確認項目:

- 現在ブランチ
- 差分の有無
- 最後に通したテスト
- 未完了タスク
- 次の 1 手

## 運用ルール

- `docs/superpowers` の文書だけを更新して終わらない
- 正式文書に未反映の内容がある場合は、`progress-summary` に明記する
- セッション終了前に `Next step` を 1 つだけ書く
- 日付付き作業文書は増えてよいが、正式文書は固定名で育てる
- 再開性は会話履歴ではなく文書と Git 状態で担保する

## 当面の実践ルール

当面は次を標準とする。

1. 新テーマ開始時に、正式文書の更新対象を先に宣言する
2. 設計書と実装計画を作るときに、同時に `progress-summary` も作る
3. 各セッション終了前に `progress-summary` を更新する
4. 節目ごとに `update-docs` を行い、`docs/` の正式文書を育てる

これにより、Superpowers の「1 テーマを前に進める強さ」を維持しつつ、複数セッション前提の開発にも耐えられる状態を作る。
