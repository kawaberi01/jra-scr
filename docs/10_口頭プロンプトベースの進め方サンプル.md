# 口頭プロンプトベースの進め方サンプル

## 目的

この文書は、これまで整理したフローを、実際にユーザーが口頭またはテキストでどのように指示するかという観点で軽く示すものである。

ここでは厳密なプロンプトテンプレートではなく、実際の会話で使いやすいレベルの指示例を並べる。

この版では、`progress-summary` はユーザーが毎回指示しなくても、フローの一部として自動的に作成・更新される前提でシミュレーションする。

## 前提

使うフローは次のとおりとする。

1. `artifact-planning`
2. `brainstorming`
3. `using-git-worktrees`
4. `writing-plans`
5. `executing-plans` / `test-driven-development`
6. `requesting-code-review`
7. `dual-track-documentation`
8. `verification-before-completion`
9. `finishing-a-development-branch`

補足:

- `progress-summary` はフロー内で自動更新される
- ユーザーは `progress-summary` を個別に指示しない
- セッション終了時や節目では、エージェント側が進捗と文書同期状況を残す

## 全体像

### 典型的な開始指示

ユーザー:

> 新しく中小規模の Web アプリを作りたいです。まず何の資料が必要か整理して、そのあと設計に進んでください。

期待される流れ:

- `artifact-planning` が起動する
- 必要資料を正式文書と作業文書に分ける
- その後 `brainstorming` に入る

## フェーズごとの会話サンプル

## 1. `artifact-planning`

### ユーザーの指示例

> タスク管理用の Web アプリを作ります。  
> まず、この規模なら何の資料が必要か決めてください。  
> GUI と API の両方があります。  
> 後で自分が見返して分かりやすい構成にしたいです。

### 期待される返答の方向

- 正式文書候補を出す
- 作業文書候補を出す
- GUI があるので UI 設計を含める
- API があるので API 仕様を含める
- `progress-summary` を最初から自動生成対象にする

### この段階の到達点

- 「何を成果物として残すか」が決まる
- まだ詳細設計には入らない

## 2. `brainstorming`

### ユーザーの指示例

> ではこのアプリの設計に入ってください。  
> まずログイン、タスク一覧、追加、完了、削除ができる構成を考えてください。  
> API と画面の責務が分かれるようにしたいです。

### 期待される返答の方向

- まず要件確認
- 複数案の提示
- 推奨案の提示
- 採用後に設計書をまとめる

### 追加で GUI を意識した指示例

> 画面遷移や主要画面の構成も整理してください。  
> 後で UI 設計書に反映できる粒度でお願いします。

### この段階の到達点

- 設計書ができる
- アーキテクチャと設計判断の元ネタが固まる

## 3. `using-git-worktrees`

### ユーザーの指示例

> 設計が固まったので、隔離された作業環境で進めてください。  
> worktree を使う前提でお願いします。

### 期待される返答の方向

- ブランチ作成
- worktree 作成
- baseline テスト確認

### この段階の到達点

- 実装を始められる状態が整う

## 4. `writing-plans`

### ユーザーの指示例

> この設計をもとに、実装計画を細かいタスクに分けてください。  
> 各タスクは 2〜5 分単位で進めやすい粒度にしてください。

### 期待される返答の方向

- 実装順序の提示
- ファイル単位の変更対象
- 検証手順
- progress-summary の初版自動作成

### この段階の到達点

- 計画と進捗管理の土台ができる

## 5. `executing-plans` / `test-driven-development`

### ユーザーの指示例

> では計画に沿って進めてください。  
> TDD で、まず認証 API から着手してください。

### 期待される返答の方向

- 失敗テスト作成
- 最小実装
- リファクタ
- progress-summary 自動更新

### GUI 実装時の指示例

> 次は一覧画面と作成フォームを進めてください。  
> UI 状態と API 連携が分かるようにしてください。

### この段階の到達点

- コードとテストが進む
- progress-summary に現在地が残る

## 6. `requesting-code-review`

### ユーザーの指示例

> ここでいったんレビューを入れてください。  
> 特に責務分離、UI/API 境界、テスト不足を見てください。

### 期待される返答の方向

- 問題点の列挙
- 重大度順の指摘
- 未解決事項の整理

### この段階の到達点

- 修正すべき論点が明確になる

## 7. `dual-track-documentation`

### ユーザーの指示例

> ここまでの内容を正式文書にも反映してください。  
> docs/superpowers だけで閉じず、docs 配下の正式文書を更新してください。

### 期待される返答の方向

- `update-docs`
- アーキテクチャ更新
- 設計判断更新
- API 仕様更新
- UI 設計更新
- 利用ガイド更新

### 軽い指示例

> 今の実装内容を docs に同期してください。

### この段階の到達点

- 作業文書の内容が正式文書へ昇格する

## 8. `verification-before-completion`

### ユーザーの指示例

> 完了扱いにする前に、必要なテストと起動確認を行ってください。  
> 実行したコマンドと結果も残してください。

### 期待される返答の方向

- テスト実行
- 起動確認
- 実行コマンドと結果の整理

### この段階の到達点

- 完了主張に根拠が付く

## 9. `finishing-a-development-branch`

### ユーザーの指示例

> このテーマを締めてください。  
> handoff が分かる状態にして、次回すぐ再開できるようにしてください。

### 期待される返答の方向

- 最終 progress-summary 自動更新
- PR またはマージ判断
- 未反映事項の有無確認

### この段階の到達点

- セッションが終わっても次回迷わない

## 軽量な一連の会話サンプル

### 開始

ユーザー:

> 新しくタスク管理 Web アプリを作ります。  
> まず必要な資料を整理して、そのあと設計に入ってください。

エージェント:

> まず成果物計画として、正式文書と作業文書を分けて必要資料を整理します。  
> GUI と API があるため、概要、アーキテクチャ、設計判断、利用ガイド、UI 設計、API 仕様、進捗メモを候補にします。

### 設計へ移行

ユーザー:

> その方針で進めてください。  
> 画面と API の責務が分かる設計にしたいです。

エージェント:

> まず主要ユースケースと画面を整理し、そのうえで 2〜3 案を比較して設計を固めます。

### 実装計画へ移行

ユーザー:

> 設計が固まったら、実装計画に落としてください。

エージェント:

> 実装を API、認証、UI 一覧、UI 操作、文書反映に分解し、各タスクの検証手順も含めて計画化します。  
> progress-summary はフローの一部として自動で初期化し、以後の節目でも更新します。

### 実装中

ユーザー:

> 認証 API から TDD で進めてください。  
> 節目ごとに docs も更新してください。

エージェント:

> まず失敗テストを書き、その後最小実装を入れます。  
> バックエンド基盤ができた段階で API 仕様と設計判断を docs に反映し、進捗要約も自動で残します。

### 終了前

ユーザー:

> 今日はここまでにします。次回すぐ再開できる状態にしてください。

エージェント:

> progress-summary に現在地、未完了、次の一手、文書反映状況、最後に通したテストを反映して終了します。

## 実際にユーザーが意識すること

ユーザーが毎回複雑な定型文を覚える必要はない。

最低限意識すればよい指示は次の程度で十分である。

1. 最初:
   - 「まず必要資料を整理してください」
2. 設計:
   - 「そのあと設計に入ってください」
3. 実装計画:
   - 「計画に落としてください」
4. 実装中:
   - 「TDD で進めて、節目で docs に反映してください」
5. 終了時:
   - 「次回すぐ再開できる状態にしてください」

## 結論

このフローは、ユーザーが長い儀式的プロンプトを毎回書かなくても運用できる形にできる。

重要なのは、ユーザーの指示が次の 3 点を含むことである。

1. 最初に必要資料を決める
2. セッション継続のための進捗記録をフローに埋め込む
3. 作業文書だけでなく正式文書へ反映する

この 3 点を入れれば、実際の会話ベースの開発フローとして十分運用可能である。
