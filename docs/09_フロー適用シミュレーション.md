# フロー適用シミュレーション

## 目的

この文書は、`08_必要資料一覧と適用フロー.md` で整理したフローを実際に使った場合、どのような順序で資料と実装が進むかを具体的に示すものである。

ここでは、個人開発の中小規模 Web アプリケーションを題材に、複数セッションにまたがる進め方をシミュレーションする。

## 題材

題材は次のような Web アプリとする。

- アプリ名: 個人用タスク管理アプリ
- 主要機能:
  - メールログイン
  - タスクの一覧表示
  - タスク追加、完了、削除
  - フィルタ
- 構成:
  - フロントエンドあり
  - バックエンド API あり
  - 認証あり

この題材は API だけでもなく、大規模エンタープライズでもないため、個人開発の中小規模案件の例として扱いやすい。

## 前提となるフロー

使うフローは次のとおりとする。

1. `artifact-planning`
2. `brainstorming`
3. `using-git-worktrees`
4. `writing-plans`
5. `executing-plans` / `subagent-driven-development`
6. `test-driven-development`
7. `requesting-code-review`
8. `dual-track-documentation`
9. `verification-before-completion`
10. `finishing-a-development-branch`

## この案件で必要と判断する資料

### 正式文書

- `docs/00_成果物戦略と開発フロー.md`
- `docs/01_プロジェクト概要.md`
- `docs/02_アーキテクチャ.md`
- `docs/03_設計判断.md`
- `docs/04_利用ガイド.md`
- `docs/05_UI設計.md`
- `docs/06_API仕様.md`
- `docs/07_セキュリティと運用注意.md`

### 作業文書

- `docs/superpowers/specs/YYYY-MM-DD-task-app-design.md`
- `docs/superpowers/plans/YYYY-MM-DD-task-app-implementation-plan.md`
- `docs/superpowers/YYYY-MM-DD-task-app-progress-summary.md`

この案件では、GUI、API、認証があるため、UI 設計、API 仕様、セキュリティ文書が必要と判断する。

## セッション 1: 成果物計画

### 使うスキル

- `artifact-planning`

### このセッションでやること

- 何を成果物として残すかを決める
- 正式文書と作業文書を分ける
- `progress-summary` の雛形を作る
- このテーマで更新対象になる正式文書を決める

### この段階で決まること

- GUI があるので `UI設計` が必要
- API を分けるので `API仕様` が必要
- 認証があるので `セキュリティと運用注意` が必要
- 初回の正式文書更新対象は次のとおり
  - `docs/01_プロジェクト概要.md`
  - `docs/02_アーキテクチャ.md`
  - `docs/05_UI設計.md`
  - `docs/06_API仕様.md`
  - `docs/07_セキュリティと運用注意.md`

### このセッションの成果物

- 正式文書:
  - 成果物戦略文書
- 作業文書:
  - progress-summary 初版

### progress-summary に残す内容

- `Current status`: 成果物計画完了
- `Completed`: 必要資料の決定
- `Remaining`: 設計、計画、実装
- `Next step`: 設計書を作成する
- `Doc sync status`: 正式文書は未作成または未更新

## セッション 2: 設計

### 使うスキル

- `brainstorming`

### このセッションでやること

- 画面、API、認証を含めた設計を固める
- 主要な設計判断を明確化する
- 実装前に決めるべきことを文書化する

### 設計で決まる内容

- フロントエンドは SPA
- バックエンドは REST API
- 認証はメール + セッション
- タスク一覧画面、作成 UI、フィルタ UI を持つ
- タスク操作は optimistic update を採用しない
- API の基本エンドポイントを定義する

### このセッションの成果物

- 作業文書:
  - 設計書

### 後で更新対象となる正式文書

- `docs/02_アーキテクチャ.md`
- `docs/03_設計判断.md`
- `docs/05_UI設計.md`
- `docs/06_API仕様.md`

### progress-summary に残す内容

- `Current status`: 設計完了
- `Completed`: 設計書作成
- `Remaining`: 実装計画、初回実装
- `Next step`: 実装計画を作成する
- `Doc sync status`: 正式文書へ未反映

## セッション 3: 実装計画

### 使うスキル

- `using-git-worktrees`
- `writing-plans`

### このセッションでやること

- ブランチと worktree を切る
- 実装タスクへ分解する
- タスク単位の検証方法を決める

### 分解されるタスク例

- Task 1: 認証 API の土台
- Task 2: タスク一覧 API
- Task 3: タスク作成 API
- Task 4: 一覧画面 UI
- Task 5: 作成フォーム UI
- Task 6: 完了・削除 UI

### このセッションの成果物

- 作業文書:
  - 実装計画
  - 更新済み progress-summary

### progress-summary に残す内容

- `Current status`: 実装計画完了
- `Completed`: task 分解、worktree 準備
- `Remaining`: 実装全般
- `Next step`: 認証 API の失敗テストを書く
- `Verification`: baseline テスト確認
- `Doc sync status`: 正式文書未反映

## セッション 4: バックエンド初回実装

### 使うスキル

- `test-driven-development`
- `executing-plans`
- `requesting-code-review`

### このセッションでやること

- 認証 API とタスク一覧 API を実装する
- テストから始める
- レビューで API 境界とモデル設計を確認する

### このセッションで更新されるもの

- コード
- バックエンドテスト
- progress-summary

### レビューで見られる点

- 認証フローが曖昧でないか
- モデル責務が膨らみすぎていないか
- API のエラー設計が揃っているか

### progress-summary に残す内容

- `Current status`: バックエンド基盤実装済み
- `Completed`: Task 1, Task 2
- `Remaining`: UI 実装、追加 API
- `Next step`: UI 一覧画面の設計テストへ進む
- `Verification`: 認証 API / 一覧 API テスト通過
- `Doc sync status`: API 仕様未反映

## セッション 5: 初回 update-docs

### 使うスキル

- `dual-track-documentation`

### このセッションでやること

- 作業文書の内容を正式文書へ反映する
- いまのコードベースに合わせて docs を更新する

### 更新する正式文書

- `docs/02_アーキテクチャ.md`
  - SPA + API 構成
- `docs/03_設計判断.md`
  - optimistic update を採用しない判断
- `docs/06_API仕様.md`
  - ログイン API
  - タスク一覧 API
- `docs/07_セキュリティと運用注意.md`
  - 認証方式
  - セッション管理上の注意

### progress-summary に残す内容

- `Doc sync status`: backend 関連は反映済み、UI 文書は未反映
- `Next step`: 一覧画面 UI を実装する

## セッション 6: フロントエンド実装

### 使うスキル

- `test-driven-development`
- `executing-plans`
- `requesting-code-review`

### このセッションでやること

- 一覧画面 UI
- 作成フォーム UI
- 完了・削除操作 UI

### この段階で重要な観点

- 画面状態
- 読み込み中表示
- エラー表示
- フォーム入力時の挙動
- 操作後のリスト更新

### review で見る点

- UI と API 契約がずれていないか
- 画面責務とコンポーネント責務が混ざっていないか
- アクセシビリティ上の問題がないか

### progress-summary に残す内容

- `Current status`: 主要 UI 完成
- `Completed`: Task 4, Task 5, Task 6
- `Remaining`: 利用ガイド整備、総合確認
- `Next step`: UI 設計と利用ガイドへ反映する
- `Verification`: UI テスト通過
- `Doc sync status`: UI 文書未反映

## セッション 7: UI の update-docs と総合検証

### 使うスキル

- `dual-track-documentation`
- `verification-before-completion`

### このセッションでやること

- UI の実装内容を正式文書へ反映する
- 利用ガイドを書く
- 動作確認とテストをまとめる

### 更新する正式文書

- `docs/04_利用ガイド.md`
  - ログイン方法
  - タスク操作方法
- `docs/05_UI設計.md`
  - 画面一覧
  - 主要画面の役割
  - ユーザーフロー

### 検証で残す内容

- 実行したテストコマンド
- 手動確認内容
- 未確認事項

### progress-summary に残す内容

- `Current status`: 完了候補
- `Completed`: 正式文書更新、主要検証
- `Remaining`: 軽微な修正のみ
- `Next step`: 完了処理または残課題修正
- `Verification`: 総合確認済み
- `Doc sync status`: 主要文書反映済み

## セッション 8: 完了処理

### 使うスキル

- `finishing-a-development-branch`

### このセッションでやること

- 最終 handoff を残す
- PR またはマージ判断をする
- 次のテーマへ切り替えられる状態にする

### 最終的に残るもの

- 正式文書一式
- 作業文書一式
- 最新の progress-summary
- コード、テスト、検証結果

## このシミュレーションで分かること

### 1. 最初に成果物計画があると迷いにくい

設計の前に、何を正式文書として残すかが決まるため、`brainstorming` が単発の設計書づくりで終わらない。

### 2. progress-summary がセッション継続の中心になる

毎回 `Current status`、`Next step`、`Doc sync status` を残すことで、別セッションでも再開しやすい。

### 3. update-docs を途中で入れると docs が育つ

最後にまとめて文書化するのではなく、実装の節目ごとに正式文書を更新することで、最終成果物が自然に整っていく。

### 4. GUI があっても同じ骨格で回せる

違いは、必要文書に `UI設計` が増え、実装の途中でそれを `update-docs` する点である。フロー自体は変わらない。

## 結論

このフローは、API だけでなく、中小規模の個人開発 Web アプリにもそのまま適用できる。

重要なのは次の 3 点である。

1. 最初に `artifact-planning` で必要資料を決める
2. セッションごとに `progress-summary` を更新する
3. 節目で `dual-track-documentation` により正式文書へ反映する

この 3 点が入ることで、1 セッションで終わらない開発でも、後で見返して分かりやすい状態を維持できる。
