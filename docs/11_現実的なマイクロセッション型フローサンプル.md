# 現実的なマイクロセッション型フローサンプル

## 目的

この文書は、全体設計のあとに「この機能を少し進めてください」「今日はここまでです」という細かいセッションが何度も続く前提で、現実的な運用サンプルを示す。

ここで強調したいのは次の点である。

- 全体設計は最初にまとめて行う
- しかし実装は一気通貫では進まない
- 1 セッションで 1 機能全部を終わらせる前提にしない
- 実際には小さなタスク単位で何度も再開する

## 前提

題材は中小規模の個人開発 Web アプリとする。

最初に次は終わっているものとする。

- `artifact-planning`
- `brainstorming`
- `writing-plans`

つまり、全体の成果物計画、全体設計、実装計画はすでにある。

この後に起きる現実的な流れをシミュレーションする。

## 重要な考え方

全体設計が終わったあと、ユーザーは毎回こういう粒度で依頼すると考える。

- ログイン API だけ進めてください
- 一覧画面だけ作ってください
- 追加フォームだけやってください
- ここまでの docs を更新してください
- 今回のレビューだけしてください

つまり、設計以降は「機能単位」よりさらに細かい「タスク単位」のループになる。

このループを支えるのが:

- 実装計画
- progress-summary
- docs への段階反映

である。

## 現実的な流れの全体像

### Phase A. 一度だけ大きくやるもの

1. `artifact-planning`
2. `brainstorming`
3. `using-git-worktrees`
4. `writing-plans`

ここで終わること:

- 必要資料が決まる
- 全体設計がある
- 実装計画がある
- 初期 progress-summary がある

### Phase B. 以後ずっと繰り返すもの

以下の小ループを何度も回す。

1. 今回やる小タスクを決める
2. TDD で実装する
3. 必要ならレビューする
4. progress-summary を更新する
5. 節目なら docs を更新する
6. セッションを閉じる

## 小ループのテンプレート

### セッション開始

ユーザー:

> 次はログイン API のセッション管理部分だけ進めてください。

エージェントが行うこと:

- 実装計画の中から該当タスクを特定する
- progress-summary から現在地を確認する
- 今回のスコープを明確化する

この時点では、新しい全体設計はしない。

### 実装

エージェントが行うこと:

- 対象タスクの失敗テストを書く
- 最小実装を入れる
- リファクタする
- 今回触った範囲だけ検証する

### セッション終了

エージェントが行うこと:

- 今回完了したことを残す
- 次回の一手を 1 つに絞る
- docs 反映が必要ならその状態を残す

ユーザーはこれを毎回指示しない。

## 具体シミュレーション

ここでは、全体設計完了後に 8 回の小セッションで進む例を示す。

## セッション 1: ログイン API の開始

ユーザー:

> ログイン API の土台だけ進めてください。

エージェント:

- 実装計画から `auth API base` タスクを拾う
- 失敗テストを書く
- 最小実装を入れる
- 認証 API の基本ルートだけ通す
- progress-summary を更新する

この回で終わるもの:

- API の骨組み

この回で終わらないもの:

- セッション永続化
- エラーハンドリング
- docs 更新

次の一手:

- セッション管理ロジックを追加する

## セッション 2: ログイン API の仕上げ

ユーザー:

> 前回の続きを進めてください。ログイン周りをもう少し完成に近づけてください。

エージェント:

- progress-summary から前回の次の一手を読む
- セッション管理とエラー処理を実装する
- 認証テストを追加する
- 必要ならコードレビューを入れる
- progress-summary を更新する

この回で終わるもの:

- ログイン API 一式

まだ終わらないもの:

- API 仕様反映
- UI 側の接続

次の一手:

- タスク一覧 API の取得処理に着手する

## セッション 3: タスク一覧 API

ユーザー:

> 次はタスク一覧 API を進めてください。

エージェント:

- 一覧 API の取得テストを書く
- 最小実装を入れる
- 認証との境界を確認する
- progress-summary を更新する

この回では docs 反映をまだしない。

理由:

- 安定した知識がまだ少ない
- API 群をある程度まとめてから反映した方がきれい

## セッション 4: API 群の docs 同期

ユーザー:

> ここまでの API を docs に反映してください。

エージェント:

- `dual-track-documentation` を使う
- `docs/06_API仕様.md` を更新する
- `docs/03_設計判断.md` に認証や API 方針を反映する
- progress-summary の `Doc sync status` を更新する

ここではコードをほとんど触らないこともある。

## セッション 5: 一覧画面 UI の骨組み

ユーザー:

> 次は一覧画面だけ進めてください。

エージェント:

- 一覧画面の UI テストまたは画面確認観点を決める
- 最小 UI を実装する
- loading / empty / error を整理する
- API 取得とつなぐ
- progress-summary を更新する

この回で終わるもの:

- 一覧画面の初版

まだ終わらないもの:

- 作成フォーム
- 完了、削除操作
- UI 設計文書反映

## セッション 6: 作成フォーム

ユーザー:

> 作成フォームだけ作ってください。

エージェント:

- フォーム入力と送信処理を実装する
- バリデーションを足す
- 一覧との接続を確認する
- progress-summary を更新する

次の一手:

- 完了・削除操作を追加する

## セッション 7: UI の見直しとレビュー

ユーザー:

> UI と API のつながりをレビューしてください。必要なら直してください。

エージェント:

- `requesting-code-review` を使う
- UI/API 境界、状態管理、責務分離を確認する
- 問題があれば局所修正する
- progress-summary を更新する

この回では、新機能を増やさない可能性もある。

## セッション 8: UI と利用ガイドの docs 同期

ユーザー:

> ここまでの内容を docs に反映してください。今日はそこで止めます。

エージェント:

- `dual-track-documentation` を使う
- `docs/05_UI設計.md` を更新する
- `docs/04_利用ガイド.md` を更新する
- progress-summary に現在地と次の一手を残す

次回の一手:

- 完了・削除操作を実装する

## このシミュレーションの特徴

### 1. 全体設計は先に終わるが、実装は分割される

設計が終わったからといって、そのまま最後まで一気に進むわけではない。

実際には:

- API の一部
- UI の一部
- docs 反映
- レビュー

のように、毎回小さな目的で区切られる。

### 2. docs 更新だけのセッションがある

現実には、コードを書かずに docs だけ更新する回も自然に発生する。

### 3. review だけのセッションもある

「今日は実装ではなくレビューだけ」という回もある。

### 4. progress-summary がループの接着剤になる

ユーザーは毎回 progress-summary を意識しないが、フロー側では毎回更新される。

これにより:

- 前回どこまで進んだか
- 今回何をやるか
- docs がどこまで反映されたか

がつながる。

## 口頭プロンプトの現実的な粒度

現実のやり取りは、次のような短い指示の繰り返しになる。

- 「ログイン API の続きだけ進めてください」
- 「一覧画面だけ作ってください」
- 「ここまでの API を docs に反映してください」
- 「レビューだけしてください」
- 「今日はここまでなので次回すぐ再開できる状態にしてください」

重要なのは、ユーザーが毎回フルフローを言わなくてもよいことだ。

フローの骨格は内部で維持され、ユーザーはそのときの小タスクだけ指示する。

## 結論

現実的な運用では、全体設計のあとに「小さな実装セッションのループ」が長く続く。

したがって、フローの見せ方も次の形が正しい。

1. 最初に大きな設計フェーズを通す
2. その後は小タスク単位の反復ループに入る
3. 各ループで progress-summary と docs 同期が裏で回る

この形であれば、1 セッションで全部終わる前提にせず、現実的な個人開発の使い方に近づけられる。
