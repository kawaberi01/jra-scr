# 協調モードと自律モードの拡張案

## 目的

この文書は、これまで整理してきたフローを壊さずに、次の 2 つの運用モードを両立させるための拡張案をまとめるものである。

- 協調モード
- 自律モード

ここでいう協調モードは、人が節目ごとに確認しながら進める運用である。自律モードは、最初に大きめの意図を与えたあと、AI がある程度まとまった範囲を継続実行する運用である。

## 背景

これまでのフローは、次の点に強い。

- 複数セッション前提で再開しやすい
- 正式文書と作業文書を分けられる
- 進捗や文書同期を残しやすい
- 小さく安全に進めやすい

一方で、長時間の自律実行を前提にすると次が不足する。

- どこまで勝手に進めてよいか
- どこで止まるべきか
- 人への確認が必要な条件
- マイルストーン単位の実行契約

したがって、既存フローを単一運用とみなすのではなく、共通基盤の上に 2 モードを載せる方が自然である。

## モード分離の基本方針

### 共通部分

どちらのモードでも、次は共通とする。

1. `artifact-planning`
2. `brainstorming`
3. `using-git-worktrees`
4. `writing-plans`
5. `dual-track-documentation`
6. `verification-before-completion`
7. `finishing-a-development-branch`

つまり、成果物計画、設計、計画、文書同期、検証、完了処理は共通土台である。

### 分岐する部分

分岐するのは、設計と実装計画のあとである。

- 協調モード:
  - 小タスクごとに人が進行を見ながら進める
- 自律モード:
  - マイルストーンごとに AI が継続実行する

## 協調モード

### 目的

人が節目ごとに判断しながら進める。

### 向いているケース

- UI/UX 判断が多い
- 要件がまだ揺れている
- 外部依存が不安定
- 仕様変更が多い
- まだ実験段階

### 実行単位

- 1 セッション = 小タスク 1 個または数個
- 例:
  - ログイン API の基礎だけ
  - 一覧画面だけ
  - docs 更新だけ
  - レビューだけ

### 特徴

- 人の確認頻度が高い
- progress-summary が細かく更新される
- docs 更新も小刻みに行う
- セッション終了のたびに次の一手を残す

### 主な流れ

1. 全体設計まで完了
2. 小タスクを選ぶ
3. TDD で実装する
4. 必要ならレビューする
5. progress-summary を更新する
6. 節目なら docs を同期する
7. 次の一手を残して終える

## 自律モード

### 目的

最初に大きめの目標を与えたあと、AI がまとまった範囲を継続的に進める。

### 向いているケース

- 要件が比較的安定している
- 設計判断がほぼ固まっている
- UI の不確実性が低い
- 技術スタックが慣れている
- 人が細かく介入しなくてもよい

### 実行単位

- 1 セッション = マイルストーン 1 個
- 例:
  - 認証 API 一式
  - タスク一覧 API + 一覧 UI
  - 作成、完了、削除の一連機能
  - 文書同期と総合検証

### 特徴

- 人の確認頻度が低い
- 途中の小タスクは AI が内部で回す
- 人はマイルストーン単位で成果を見る
- progress-summary は裏で継続更新される

### 追加で必要なもの

自律モードでは、通常の設計と計画に加えて次を明示する必要がある。

1. 自律実行範囲
2. 停止条件
3. 人に確認すべき条件
4. マイルストーン
5. 各マイルストーンの完了条件

## 自律実行契約

自律モードでは、実装に入る前に「自律実行契約」を定義する。

### 契約で決めること

#### 1. 進めてよい範囲

例:

- 認証 API は設計書の範囲内で自由に実装してよい
- 一覧 UI は既存 UI 方針に従う限り自由に実装してよい

#### 2. 止まるべき条件

例:

- DB スキーマを大きく変える必要が出た
- UI 方針を根本から変える必要が出た
- セキュリティ判断が必要になった
- 外部 API 制約が想定と違った
- 設計書にない機能追加が必要になった

#### 3. 人に確認する条件

例:

- UX で複数案がある
- 認証方式を変更する必要がある
- スクレイピング対象の制約が厳しい
- 既存データを破壊しうる変更がある

#### 4. マイルストーン

例:

- Milestone 1: 認証 API 完了
- Milestone 2: 一覧 API + 一覧 UI 完了
- Milestone 3: 作成、完了、削除完了
- Milestone 4: docs 同期 + 総合検証

## フローの拡張形

### 協調モードの拡張フロー

1. `artifact-planning`
2. `brainstorming`
3. `using-git-worktrees`
4. `writing-plans`
5. 小タスク反復ループ
   - `test-driven-development`
   - `executing-plans`
   - `requesting-code-review`
   - `dual-track-documentation`
6. `verification-before-completion`
7. `finishing-a-development-branch`

### 自律モードの拡張フロー

1. `artifact-planning`
2. `brainstorming`
3. `using-git-worktrees`
4. `writing-plans`
5. 自律実行契約を定義
6. マイルストーン反復ループ
   - 小タスクを内部で自律実行
   - マイルストーン完了時にレビュー
   - `dual-track-documentation`
7. `verification-before-completion`
8. `finishing-a-development-branch`

## 2 モードを両立させる考え方

### 設計までは共通

最初の大枠は共通でよい。

- 何の資料が必要か
- どの設計で進むか
- どの文書を更新するか

これは協調でも自律でも必要である。

### 実装以降でモードを選ぶ

違いが出るのは実装以降である。

- 人が細かく関わるなら協調モード
- AI にまとめて進めさせるなら自律モード

つまり、モードの切り替え点は `writing-plans` のあとに置くのが自然である。

## どちらを選ぶべきか

### 協調モードを選ぶべきケース

- プロダクトの方向性がまだ不確定
- UI の正解が見えていない
- 要件が毎回変わる
- 設計より実験が多い
- 人が判断したいことが多い

### 自律モードを選ぶべきケース

- 設計が固まっている
- 実装の繰り返し性が高い
- 人が途中で毎回見なくてもよい
- マイルストーン単位で成果を見れば足りる

## progress-summary の位置づけ

### 協調モード

- 1 セッションごとに細かく更新される
- 小さな現在地を残す

### 自律モード

- 内部では継続更新する
- 人が見るのは主にマイルストーン時点

つまり、progress-summary は両方で必要だが、見せ方が違う。

## docs 同期の位置づけ

### 協調モード

- 節目ごとに小刻みに更新する

### 自律モード

- マイルストーン完了時にまとめて更新する

これにより、自律モードでも docs が最後まで放置されないようにする。

## 実際の使い分け例

### 例 1. GUI が多い新規 Web アプリ

- 初期は協調モード
- 画面方針が固まったら一部を自律モードへ移す

### 例 2. API 中心の管理画面バックエンド

- 設計確定後は自律モードに向く

### 例 3. スクレイピングや外部依存が強い案件

- 不確実性が高いので協調モード寄り
- 安定部分のみ自律モードへ切り出す

## 今後のスキル拡張案

この 2 モードを本当に運用するなら、追加候補は次のとおりである。

- `addon-superpowers-artifact-planning`
- `addon-superpowers-dual-track-documentation`
- `addon-superpowers-autonomous-execution-contract`

最後のスキルは、自律モード専用に次を決める役割を持つ。

- マイルストーン
- 自律許可範囲
- 停止条件
- 人確認条件

## 結論

既存のフローは、協調モードの土台としてかなり強い。

ただし、長時間自律実行を本格的に扱うには、次を追加する必要がある。

- 自律実行契約
- マイルストーン運用
- 停止条件

したがって、最も現実的な拡張は次の方針である。

1. 設計までは共通
2. 実装以降で協調モードと自律モードを分ける
3. progress-summary と docs 同期は両モードで維持する

この構成なら、人が細かく関与する運用と、より長く自律実行させる運用の両方を同じ土台の上で扱える。
