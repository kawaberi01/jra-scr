# 簡単な回し方

## 目的

この文書は、Human in the Loop フローを実際に回すときの最小限の使い方を、迷わない形でまとめたものである。

対象は、個人開発の小規模から中小規模のアプリケーションである。

## 前提

この回し方は、次のスキルを前提にする。

- `addon-superpowers-artifact-planning`
- `brainstorming`
- `writing-plans`
- `addon-superpowers-hitl-execution`
- `addon-superpowers-auto-progress`
- `addon-superpowers-dual-track-documentation`

## 基本の考え方

- 最初に全体の成果物と設計を決める
- 実装は一気通貫でやらない
- 1 セッションでは小さい範囲だけ進める
- 毎回 `progress-summary` を起点に続きから入る
- 節目で docs に反映する

## 最初にやること

### 1. 必要資料を決める

ユーザーの指示例:

> まず必要資料を整理してください。

ここで決めるもの:

- 正式文書
- 作業文書
- `progress-summary`

### 2. 設計する

ユーザーの指示例:

> そのあと設計に入ってください。

ここで作るもの:

- 設計書

### 3. 実装計画に落とす

ユーザーの指示例:

> 設計が固まったら実装計画に落としてください。

ここで作るもの:

- 実装計画
- 初期 `progress-summary`

## 実装フェーズの回し方

ここからは短いセッションを何度も回す。

### 毎回の開始

ユーザーの指示例:

> 続きから進めてください。

エージェントがやること:

1. design を読む
2. plan を読む
3. `progress-summary` を読む
4. 今回の小タスクを 1 つ決める
5. 終了条件を決める

### 毎回の実装

ユーザーの指示例:

> 一覧 API だけ進めてください。

または

> 追加フォームだけ作ってください。

エージェントがやること:

1. 小さい範囲だけ進める
2. 必要なら TDD で進める
3. テストまたは起動確認を行う
4. `progress-summary` を更新する

### 毎回の終了

ユーザーの指示例:

> 今日はここまでなので次回すぐ再開できるようにしてください。

エージェントがやること:

1. 完了したことを残す
2. 未完了を残す
3. `Next Step` を 1 つだけ残す
4. `Verification` を残す
5. `Doc Sync Status` を残す

## docs を更新するタイミング

毎回更新する必要はない。

次のときに更新する。

- API の仕様が安定した
- UI の動きが安定した
- 設計判断が固まった
- 利用方法が固まった

ユーザーの指示例:

> ここまでの内容を docs に反映してください。

## この回し方でやらないこと

- 1 セッションで全部終わらせようとしない
- 毎回フルフローを説明しない
- docs を毎回全部更新しない
- `progress-summary` をユーザーが手で書かない

## 実際の最小パターン

1. `必要資料を整理してください`
2. `そのあと設計に入ってください`
3. `実装計画に落としてください`
4. `続きから進めてください`
5. `この機能だけ進めてください`
6. `ここまでの内容を docs に反映してください`
7. `今日はここまでなので次回すぐ再開できるようにしてください`

この 7 つで十分回る。

## 回すうえでの注意

- 1 セッションのスコープは小さくする
- `Next Step` は 1 つにする
- `progress-summary` は並列更新しない
- docs 更新は安定した内容だけにする

## 結論

簡単に回すなら、次の 3 点だけ守ればよい。

1. 最初に成果物と設計を決める
2. 実装は小さいセッションで進める
3. 毎回 `progress-summary` を起点に続きから入る

これで、Human in the Loop フローは実用的に回せる。
