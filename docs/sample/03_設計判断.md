# 設計判断

この文書は、主要な設計上の選択について「なぜそうしたか」をまとめたものです。  
各判断には、検討した代替案と、採用した理由を記載します。

## 1. 直接 API ではなく CLI サブプロセスを使う

**判断:** Anthropic API を直接呼ぶのではなく、`claude -p --output-format stream-json` をサブプロセスとして起動する。

**検討した代替案:**
- Anthropic API を直接呼ぶ
- Claude Code をライブラリとして組み込む

**CLI を採用した理由:**
- `CLAUDE.md`、skills、tools、memory、MCP servers、hooks などをそのまま活用できる
- Claude Code に新機能が追加されても、ブリッジ側の追従実装がほぼ不要
- ツール実行、権限処理、エラー回復などの複雑な挙動を CLI 側に任せられる
- このプロジェクトは Claude Code の代替品ではなく、Claude Code の UI である

**受け入れたトレードオフ:**
- サブプロセス起動分の遅延がある
- セッションごとに OS プロセスを使うためリソース消費が増える
- 構造化済みレスポンスではなく `stream-json` を解析する必要がある

## 2. Thread = Session の 1:1 対応

**判断:** Discord の 1 スレッドを、Claude Code の 1 セッションに対応付ける。

**検討した代替案:**
- 全スレッドで 1 つの共有セッションを使う
- ユーザーごとにセッションを分ける
- チャンネルごとにセッションを分ける

**採用理由:**
- Discord ユーザーにとって「スレッド = 1つの話題」は自然な UX
- スレッドを上にたどるだけで会話履歴を追える
- `--resume session_id` で Claude Code の内部状態ごと継続できる
- スレッドごとに別作業を並列で進められる

**実装方針:** SQLite の `sessions` テーブルで `thread_id` を主キー、`session_id` を Claude Code 側の識別子として保存する。

## 3. 共有実行ヘルパー `_run_helper.py`

**判断:** Claude CLI 実行ロジックを `_run_helper.py` に集約し、`ClaudeChatCog` と `SkillCommandCog` の両方から使う。

**検討した代替案:**
- 各 Cog に同じストリーミング処理を重複実装する
- 継承ベースの共通基底クラスにまとめる
- pub/sub 的なイベント構成にする

**採用理由:**
- DRY を守りつつ、入力経路の違いは各 Cog に残せる
- ステータス表示、メッセージ分割、エラー処理を 1 か所で直せる
- 将来の Cog 追加時も同じ関数を呼べばよい

## 4. ステータス表示に絵文字リアクションを使う

**判断:** ユーザーの元メッセージに絵文字リアクションを付けて、現在の状態を示す。

**検討した代替案:**
- スレッド内に専用ステータスメッセージを出す
- typing indicator だけ使う
- 埋め込みを頻繁に更新する

**採用理由:**
- スレッドを状態通知で汚しにくい
- 一目で状況が分かる
- メッセージ編集より API コストが低い
- Discord モバイルでも見やすい

**実装上のポイント:**
- 急な切り替えでレート制限に当たらないよう 700ms デバウンス
- 同時に表示するステータス絵文字は 1 つだけ
- 一定時間動きがなければ stall 検知を出す

## 5. コードフェンスを壊さないメッセージ分割

**判断:** Discord 文字数制限で分割するとき、コードフェンスの途中で切らない。

**検討した代替案:**
- 2000 文字で機械的に切る
- 段落単位だけで切る
- 画像にして送る

**採用理由:**
- Claude Code は大量のコードブロックを出力する
- フェンス途中の分割は見た目が悪く、可読性も下がる
- 正しく閉じて開き直せば、次のメッセージでもシンタックスハイライトを保てる

**優先する分割順:**
1. 段落境界
2. 行境界
3. 最後の手段としての強制分割

## 6. 単一の巨大 Bot ではなく、インストール可能なパッケージにする

**判断:** `claude_discord` を Python パッケージとして配布する。

**検討した代替案:**
- モノリシックな単一 Bot として提供する
- Docker イメージだけを配る
- コピーして使うテンプレートにする

**採用理由:**
- フレームワーク部分と利用者のカスタムコードを分離できる
- パッケージ更新で本体改善を取り込みやすい
- 利用側プロジェクトごとにバージョン固定できる
- 実際に EbiBot でこの運用が成立している

## 7. ブリッジ自身には AI ロジックを持たせない

**判断:** ブリッジには独自の AI ロジック、独自のシステムプロンプト、独自メモリを持たせない。

**検討した代替案:**
- Discord 向け専用システムプロンプトを追加する
- 独自ツールを定義する
- 独自のメモリ / コンテキスト管理を持つ

**採用理由:**
- それらは本来 Claude Code 側が担うべき責務
- 独自ロジックはメンテナンス負債になりやすい
- ターミナルでの挙動と Discord 上の挙動を一致させやすい
- 振る舞いの設定元を `CLAUDE.md` に一本化できる

## 8. セッション保存は SQLite

**判断:** スレッドとセッションの対応は SQLite に保存する。

**検討した代替案:**
- メモリ上の辞書
- Redis
- PostgreSQL
- JSON ファイル

**採用理由:**
- Bot 再起動後も継続できる
- 追加インフラが不要
- `aiosqlite` でイベントループを塞がず扱える
- スキーマが単純で十分

**接続方針:** コネクションプールは持たず、各操作ごとに接続を開いて閉じる。効率は少し落ちるが、構成が単純でこの規模には十分。

## 9. Runner インスタンスは clone パターンで複製する

**判断:** `ClaudeRunner.clone()` で設定を引き継いだ新しい runner を作る。

**検討した代替案:**
- 同じ runner インスタンスを使い回す
- 毎回 `ClaudeRunner` を手で new する
- 専用 factory を大きく組む

**採用理由:**
- セッションごとのプロセス状態を完全に分離できる
- 既存設定を簡単に引き継げる
- `thread_id -> runner` の追跡がしやすい

## 10. Webhook と通常チャットは別経路で扱う

**判断:** Webhook メッセージと通常ユーザーのメッセージは、別の Cog / 判定経路で扱う。

**検討した代替案:**
- 1 つの Cog で両方扱う
- ミドルウェア的な前処理を作る

**採用理由:**
- 責務が混ざらない
- セキュリティ条件が異なる
- 自動ワークフローと対話型セッションの振る舞いを分けやすい
- 判定条件が排他的なので衝突しにくい

## 11. Status Manager に stall 検知を持たせる

**判断:** 一定時間ツール動作がない場合、⏳ や ⚠️ を表示する。

**採用理由:**
- Claude Code は長く考えることがあり、止まったのか思考中なのか見分けにくい
- ストリーム停止やネットワーク問題の兆候に早く気付ける
- 「少し遅い」と「異常かもしれない」を段階表示で区別できる
- 非同期タスクで実装でき、他の I/O を妨げない
