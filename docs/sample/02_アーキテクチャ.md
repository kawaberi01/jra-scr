# アーキテクチャ

## 概要

`claude-code-discord-bridge` は、Discord のメッセージと Claude Code CLI をつなぐ薄い UI レイヤーです。  
このプロジェクト自体は AI の判断ロジックを持たず、知能の本体は Claude Code 側の機能
(`CLAUDE.md`、skills、tools、memory、MCP servers など) にあります。

このブリッジの役割は次の 4 つです。
- Discord から入力を受け取る
- CLI を起動する
- CLI の出力を解析する
- 結果を Discord に見やすく返す

## モジュールの役割

### エントリポイント

| モジュール | 役割 |
|------------|------|
| `main.py` | 単体起動用の入口です。`.env` を読み込み、DB を初期化し、各コンポーネントを生成して Bot を起動します。 |
| `__init__.py` | 公開 API の窓口です。`ClaudeChatCog`、`ClaudeRunner`、`SessionRepository` など、組み込み利用に必要な型を外部へ公開します。 |
| `bot.py` | `ClaudeDiscordBot` を定義します。`commands.Bot` の最小ラッパーで、intents、`channel_id`、slash command 同期などを担当します。 |

### Cogs 層 (`cogs/`)

| モジュール | クラス / 関数 | 役割 |
|------------|---------------|------|
| `claude_chat.py` | `ClaudeChatCog` | 中核のメッセージ処理です。設定チャンネルとその子スレッドの `on_message` を受け、スレッド作成、セッション再開、同時実行制御を行います。 |
| `skill_command.py` | `SkillCommandCog` | `/skill` と `/skills` を提供します。`~/.claude/skills/` を読み込み、補完候補を出し、最終的な実行は `_run_helper` に委譲します。 |
| `_run_helper.py` | `run_claude_in_thread()` | 共有実行ヘルパーです。ストリーミング、ステータス更新、ツール埋め込み、エラー処理、セッション保存をまとめて扱います。 |

### Claude CLI 層 (`claude/`)

| モジュール | クラス / 関数 | 役割 |
|------------|---------------|------|
| `runner.py` | `ClaudeRunner` | サブプロセス管理です。CLI 引数の生成、環境変数の整形、`create_subprocess_exec` による起動、標準出力の逐次読取、タイムアウトや終了処理を担当します。 |
| `parser.py` | `parse_line()` | `stream-json` の 1 行を解析して `StreamEvent` に変換します。 |
| `types.py` | `StreamEvent`、`ToolUseEvent`、`SessionState` など | イベント型、ツール分類、セッション状態などの共通型を定義します。 |

### データベース層 (`database/`)

| モジュール | クラス / 関数 | 役割 |
|------------|---------------|------|
| `models.py` | `init_db()` | SQLite スキーマ定義と初期化です。`sessions` テーブルなどを作成します。 |
| `repository.py` | `SessionRepository` | セッションの CRUD を担当します。`get()`、`save()`、`delete()`、`cleanup_old()` などを提供します。 |

### Discord UI 層 (`discord_ui/`)

| モジュール | クラス / 関数 | 役割 |
|------------|---------------|------|
| `status.py` | `StatusManager` | 元メッセージへのリアクションで進行状態を表現します。ツール実行中、思考中、停止などを表します。 |
| `chunker.py` | `chunk_message()` | Discord の文字数制限に合わせて長文を安全に分割します。コードフェンスを壊さないように処理します。 |
| `embeds.py` | `tool_use_embed()`、`session_start_embed()` など | 各種埋め込みメッセージを生成します。 |

### ユーティリティ (`utils/`)

| モジュール | 関数 | 役割 |
|------------|------|------|
| `logger.py` | `setup_logging()` | ログ出力形式を設定し、`discord.py` の過剰なログを抑制します。 |

## データフロー

### 新しい会話

```text
1. ユーザーが設定チャンネルにメッセージを送る
   │
2. ClaudeChatCog.on_message()
   │
3. _handle_new_conversation()
   ├── Discord スレッドを作成
   │
4. _run_claude()
   ├── semaphore を確認
   ├── StatusManager を作成
   ├── runner を clone
   │
5. run_claude_in_thread()
   ├── SessionState を作成
   │
6. runner.run(prompt, session_id=None)
   ├── CLI 引数を構築
   ├── 環境変数を整形
   ├── create_subprocess_exec() で起動
   │
7. stream-json イベントを順次処理
   ├── SYSTEM      → session_id 保存、開始通知
   ├── ASSISTANT   → テキスト蓄積、ツール実行表示
   ├── USER        → ツール結果表示
   ├── RESULT      → 最終結果送信、完了処理
   │
8. 後始末
   ├── サブプロセス終了
   ├── リアクション掃除
   └── session_id を返す
```

### スレッド返信による会話継続

```text
1. ユーザーが既存スレッドに返信
   │
2. _handle_thread_reply()
   ├── repo.get(thread_id) で session_id を取得
   │
3. runner.run(prompt, session_id=existing_id)
   ├── --resume existing_id を付けて再開
   │
4. 新規会話と同じイベント処理を実行
```

### スキル実行

```text
1. ユーザーが /skill goodmorning を実行
   │
2. SkillCommandCog.run_skill()
   ├── スキル名を検証
   ├── 読み込み済み一覧から検索
   ├── スレッドを作成
   │
3. run_claude_in_thread(prompt="/goodmorning", session_id=None)
   │
4. 通常会話と同じ実行フローに入る
```

## 同時実行モデル

```text
                    ┌──────────────────┐
                    │   Semaphore(N)   │  N = MAX_CONCURRENT_SESSIONS
                    └────────┬─────────┘
                             │
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
   │ Thread #1   │   │ Thread #2   │   │ Thread #3   │
   │ Runner (A)  │   │ Runner (B)  │   │ Runner (C)  │
   │ claude proc │   │ claude proc │   │ claude proc │
   └─────────────┘   └─────────────┘   └─────────────┘
```

- 各セッションは `runner.clone()` により独立したプロセスを持ちます。
- `_active_runners` は `thread_id -> runner` の対応を保持し、`/clear` などで正しいプロセスを停止できます。
- `Semaphore` により同時実行数を制限し、過負荷を避けます。
- I/O は `asyncio` と `aiosqlite` ベースで非同期処理されます。

## 拡張ポイント

### フレームワーク利用者向け

1. `ClaudeChatCog` や `SkillCommandCog` を自分の `commands.Bot` に追加できます。
2. `ClaudeRunner` の `command`、`model`、`permission_mode`、`allowed_tools` などを差し替えられます。
3. `parse_line` や `chunk_message` のみを個別利用できます。
4. `run_claude_in_thread()` を自作 Cog から直接利用できます。

### フレームワーク開発者向け

1. 新しい Cog を追加する場合は `CONTRIBUTING.md` の流儀に従います。
2. 新しいツール分類を追加する場合は `types.py`、`status.py`、`embeds.py` を更新します。
3. 新しいイベント型を追加する場合は `MessageType`、`parser.py`、`_run_helper.py` を更新します。
4. 新しい埋め込み表示を追加する場合は `embeds.py` を拡張します。

## 依存関係の考え方

```text
claude_discord/
  __init__.py ──────────┬──→ claude/runner.py
                        ├──→ claude/parser.py
                        ├──→ claude/types.py
                        ├──→ cogs/claude_chat.py ──→ _run_helper.py
                        ├──→ cogs/skill_command.py ──→ _run_helper.py
                        ├──→ database/repository.py
                        ├──→ discord_ui/status.py
                        ├──→ discord_ui/chunker.py
                        └──→ discord_ui/embeds.py

  main.py ──→ bot.py, runner, claude_chat, models, repository, logger
```

重要な設計制約は、`claude/` と `discord_ui/` が直接依存しないことです。  
CLI の出力と Discord 表示が交わるのは `cogs/` 層、特に `_run_helper.py` だけです。  
この分離により、パーサは Discord をモックせずにテストでき、UI はサブプロセスをモックせずにテストできます。
