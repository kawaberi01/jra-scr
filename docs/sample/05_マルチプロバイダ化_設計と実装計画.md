# CCDB マルチプロバイダ化設計（Claude 依存の段階的解消）

## 1. 背景と目的
- 対象: Claude Code Discord Bridge (CCDB)
- 目的: Claude 固定実装を、将来的に `Claude / Codex / Copilot CLI / Gemini CLI` を選択可能な構成へ移行する
- 優先順位: まず `codex` を第一実装として追加し、既存の CCDB 体験を壊さずに拡張する

## 2. 必須要件と設計方針

### 要件1: プロバイダ抽象化
- 実行エンジンを `provider` で切替可能にする
- 初期実装は `codex` を優先

方針:
- `claude_discord/providers/` に共通インターフェースを導入
- 既存 `ClaudeRunner` は `providers/claude` 実装として再配置（または互換ラッパー化）
- `run_helper` と `event_processor` は「正規化済みイベント」のみ扱う

### 要件2: Discordスレッド ↔ セッション管理維持
- 既存の thread 単位会話継続 (`--resume`) の UX を維持

方針:
- DB の `sessions` に `provider` 列を追加し、`(thread_id -> provider + provider_session_id)` を保存
- 現行 `session_id` は後方互換のため維持しつつ、意味を「provider 側セッションID」と定義

### 要件3: 並行実行安全性維持
- worktree 分離、破壊操作前確認、セッション間共有（AI Lounge）を維持

方針:
- `SessionRegistry` / `WorktreeManager` / `Lounge` はプロバイダ非依存層として保持
- 注入テキスト内の文言を「Claude 固有」から「Assistant 汎用」へ変更

### 要件4: 設定差し替え
- 例: `ASSISTANT_PROVIDER=codex|claude|copilot|gemini`
- コマンド・パス・セッション保存先を環境変数で切替

方針:
- 新しい汎用 ENV を追加し、既存 `CLAUDE_*` は互換読み取り
- 優先順位: `PROVIDER固有` > `ASSISTANT共通` > `CLAUDE_*（互換）` > デフォルト

## 3. 提案アーキテクチャ

### 3.1 新規モジュール構成
```text
claude_discord/
  providers/
    base.py                 # ProviderProtocol, ProviderEvent, Capabilities
    registry.py             # provider 名 -> 実装の解決
    claude/
      provider.py           # 現行 ClaudeRunner ベース
      adapter.py            # stream-json -> ProviderEvent
    codex/
      provider.py           # Codex CLI 実装
      adapter.py            # codex 出力 -> ProviderEvent
```

### 3.2 ProviderProtocol（要点）
```python
class AssistantProvider(Protocol):
    name: Literal["claude", "codex", "copilot", "gemini"]
    capabilities: ProviderCapabilities

    def clone(...): ...
    async def run(prompt: str, session_id: str | None) -> AsyncGenerator[ProviderEvent, None]: ...
    async def interrupt() -> None: ...
    async def inject_tool_result(request_id: str, data: dict) -> None: ...
```

ポイント:
- `run()` は既存 `EventProcessor` が理解できるイベント型に正規化して返す
- 機能差（例: plan approval, elicitation など）は `capabilities` で判定し、UI を段階対応

### 3.3 正規化イベント（ProviderEvent）
- `message_type`: system / assistant / user / progress / result
- `text`, `tool_use`, `tool_result`, `session_id`, `is_complete`, `error`
- オプション: `permission_request`, `elicitation`, `todo_list`, `rate_limit_event`

方針:
- 最低限の共通イベント集合を定義
- プロバイダ固有イベントは `raw` に保持し、必要時のみ拡張

## 4. セッション永続化設計

### 4.1 DB変更（マイグレーション）
`sessions` テーブルへ以下を追加:
- `provider TEXT NOT NULL DEFAULT 'claude'`
- （任意）`provider_metadata TEXT` JSON

補足:
- 既存 `session_id` はそのまま利用（意味を provider 側 session_id に一般化）
- 既存データはすべて `provider='claude'` として扱う

### 4.2 Repository変更
- `save(..., provider: str = "claude")`
- `get(thread_id)` は `provider` を返す
- `get_by_session_id` は `provider` 条件付き検索を追加可能にする

## 5. 設定（ENV）設計

## 5.1 新規 ENV
- `ASSISTANT_PROVIDER=claude|codex|copilot|gemini`
- `ASSISTANT_COMMAND=...`
- `ASSISTANT_MODEL=...`
- `ASSISTANT_WORKING_DIR=...`
- `ASSISTANT_TIMEOUT_SECONDS=...`
- `ASSISTANT_ALLOWED_TOOLS=...`
- `ASSISTANT_SESSIONS_PATH=...`（provider ローカルセッション参照用）

### 5.2 provider 固有上書き
- `CLAUDE_COMMAND`, `CODEX_COMMAND`, `COPILOT_COMMAND`, `GEMINI_COMMAND`
- `CLAUDE_MODEL`, `CODEX_MODEL`, ...

### 5.3 互換性
- 既存 `.env` の `CLAUDE_*` だけでも動作継続
- `ASSISTANT_PROVIDER` 未指定時は `claude` を既定値にする

## 6. 実装フェーズ（段階移行）

### Phase 0: 互換レイヤー追加（動作不変）
- `providers/base.py`, `providers/registry.py` を追加
- `ClaudeRunner` を `AssistantProvider` 互換ラッパーで包む
- 既存テストを全通し（回帰なし）

### Phase 1: 設定切替導入
- `main.py`/`setup.py` の runner 生成を provider factory 経由へ変更
- `ASSISTANT_PROVIDER` 読み込み追加
- `run_config` の `claude_command` 等を中立名へ変更（例: `assistant_command`）

### Phase 2: DB に provider を追加
- マイグレーション追加
- `SessionRepository` 拡張
- `/resume-info` 等の表示を provider 別コマンド表記へ変更

### Phase 3: Codex provider 実装（最優先）
- `providers/codex/provider.py` と adapter 実装
- 最小機能: 新規セッション / 継続 / ストリーミング / interrupt / エラー表示
- 未対応機能は capability で明示し、UI 側で graceful degrade

### Phase 4: Claude 固有命名の整理
- `ClaudeChatCog` 等の命名は互換維持しつつ、内部は中立APIを利用
- 将来版で `AssistantChatCog` へ段階改名（deprecated 期間を設定）

## 7. Codex優先実装時の注意点
- Codex CLI の出力形式差分は adapter で吸収し、Cog 層に持ち込まない
- `session_id` の継続方式が Claude と異なる場合:
  - provider 実装内で解決（内部 state file / native resume など）
  - DB 契約は `provider + session_id` のまま統一

## 8. 既存機能への影響評価
- Thread 単位会話継続: 維持（DB契約拡張のみ）
- 並行実行（semaphore + registry）: 維持
- worktree cleanup: 維持（provider 非依存）
- AI Lounge: 維持（注入テキストを汎用化）

## 9. テスト戦略
- 既存テストを温存し、provider 選択に対するパラメタライズを追加
- 新規追加:
  - `tests/test_provider_registry.py`
  - `tests/test_provider_claude_adapter.py`
  - `tests/test_provider_codex_adapter.py`
  - `tests/test_main_provider_config.py`
  - `tests/test_session_repository_provider_column.py`
- 回帰重点:
  - thread reply の resume 継続
  - `/stop`, `/clear`, `/resume-info`
  - worktree cleanup の安全不変条件

## 10. リスクと対策
- リスク: provider ごとのイベント差分で UI 分岐が増える
  - 対策: ProviderEvent を最小共通集合に固定し、差分は capability + raw に隔離
- リスク: ENV 増加による設定複雑化
  - 対策: 優先順位を明文化し、起動時ログで最終解決値を表示
- リスク: 既存利用者の破壊的変更
  - 対策: `CLAUDE_*` 完全互換、デフォルト provider を `claude` 維持

## 11. 受け入れ基準（Done定義）
- `ASSISTANT_PROVIDER=claude` で現行テスト群が回帰なし
- `ASSISTANT_PROVIDER=codex` で以下が動作:
  - 新規スレッド開始
  - スレッド返信で継続
  - 並行実行時に競合抑止文脈（registry/lounge/worktree）が注入される
- `SessionRepository` が provider を保持し、`/resume-info` が provider 別コマンドを表示
