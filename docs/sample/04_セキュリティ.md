# セキュリティ

## 脅威モデル

`claude-code-discord-bridge` は、Claude Code CLI をサブプロセスとして起動します。  
その結果、ホストマシン上でファイル読み書き、コマンド実行、ネットワークアクセスが可能になります。  
これは設計ミスではなく、Claude Code の価値そのものです。

このブリッジのセキュリティ目標は次のとおりです。

> **許可されたユーザーだけが Claude Code セッションを起動できるようにし、ブリッジ層が Claude Code CLI 以上の新しい攻撃面を持ち込まないこと。**

### 防ぎたいこと

| 脅威 | 対策 |
|------|------|
| 未承認ユーザーが Claude を起動する | `allowed_user_ids` による許可リスト制御 |
| プロンプト経由のシェルインジェクション | `create_subprocess_exec` を使い、`shell=True` を使わない |
| プロンプト経由のフラグ注入 | `--` セパレータでユーザー入力をオプション解釈させない |
| 細工した session_id による乗っ取り | 厳密な正規表現で session_id を検証 |
| skill 名注入 | `^[\\w-]+$` の検証 |
| Bot の秘密情報漏えい | `_STRIPPED_ENV_KEYS` で環境変数をサブプロセスから除外 |
| Claude が Bash ツールで Discord トークンを読む | 環境変数を剥がしておく |
| Claude Code のネスト検知回避 | `CLAUDECODE` を除去して新規トップレベル起動にする |

### 防がないもの

| 事象 | 理由 |
|------|------|
| Claude Code がホストのファイルへアクセスすること | それが本来の機能だから |
| Claude Code がネットワーク通信すること | Web 検索や API 呼び出しに必要だから |
| Claude Code が自分の設定を変えること | `CLAUDE.md` や memory の更新は想定内だから |
| Discord サーバー管理者による乱用 | 管理者権限がある時点で別の統制が必要だから |
| 物理アクセスされたホストの防御 | これは通常のサーバー運用の範囲だから |

要点は、**セキュリティ境界は CLI ではなく Discord 側にある**ということです。  
一度セッションが始まれば、Claude Code は CLI と同等の権限を持ちます。

## 入力検証

### プロンプトの扱い (`runner.py`)

```python
# すべての引数をリストで渡し、シェル展開を行わない
args = [self.command, "-p", "--output-format", "stream-json", ...]

# ユーザーの prompt は -- 以降に置き、フラグとして解釈させない
args.append("--")
args.append(prompt)

# shell=True は使わない
self._process = await asyncio.create_subprocess_exec(*args, ...)
```

この設計で防げること:
- `--dangerously-skip-permissions` のような入力がフラグとして解釈されない
- `$(rm -rf /)` のような文字列がシェル展開されない
- 引数はそのまま `exec` に渡される

### session_id の検証 (`runner.py`)

```python
if not re.match(r"^[a-f0-9\-]+$", session_id):
    raise ValueError(f"Invalid session_id format: {session_id!r}")
```

`session_id` は Claude Code CLI から受け取り、SQLite に保存した後で `--resume` に再利用します。  
そのため、再利用前に厳密な形式検証を入れています。

### skill 名の検証 (`skill_command.py`)

```python
if not re.match(r"^[\w-]+$", name):
    await interaction.response.send_message(f"Invalid skill name: `{name}`", ephemeral=True)
    return
```

skill 名は `/{name}` という形で Claude Code に渡されるため、英数字・アンダースコア・ハイフンのみに制限しています。

## 環境変数の分離

### サブプロセスに渡さない環境変数 (`runner.py`)

```python
_STRIPPED_ENV_KEYS = frozenset({
    "CLAUDECODE",
    "DISCORD_BOT_TOKEN",
    "DISCORD_TOKEN",
    "API_SECRET_KEY",
})
```

除外理由:
1. `DISCORD_BOT_TOKEN` / `DISCORD_TOKEN`: Claude が Discord トークンを読めないようにする
2. `CLAUDECODE`: Claude Code のネスト検知を避ける
3. `API_SECRET_KEY`: REST API を使う場合の秘密鍵を渡さない

### あえて除外しないもの

`PATH`、`HOME`、`ANTHROPIC_API_KEY` などの一般的な環境変数はそのまま渡します。  
Claude Code 自体が動くために必要だからです。  
もし使用 API キーを制御したいなら、ブリッジではなく Claude Code 側の設定で行うべきです。

## 認可モデル

### ユーザー単位の認可

```python
class ClaudeChatCog(commands.Cog):
    def __init__(self, ..., allowed_user_ids: set[int] | None = None):
        self._allowed_user_ids = allowed_user_ids

    async def on_message(self, message):
        if message.author.bot:
            return
        if self._allowed_user_ids is not None and message.author.id not in self._allowed_user_ids:
            return
```

- `allowed_user_ids` が設定されている場合は、そのユーザーだけが利用可能
- `None` の場合は、チャンネル権限に委ねる
- 同じ考え方が `SkillCommandCog` にも適用される

### チャンネル単位の認可

各 Cog は、設定されたチャンネルとその子スレッドだけに反応します。  
その他のチャンネルは無視します。

### Bot メッセージの無視

`message.author.bot` を確認し、Bot や webhook のメッセージで再帰起動しないようにしています。

## Webhook セキュリティ

Webhook を受ける独自 Cog を作る場合は、次のようなパターンを守る前提です。

```python
# webhook のみ対象
if not message.webhook_id:
    return

# 固定トリガーだけを許可
if message.content.strip() != "expected-trigger":
    return

# 実行する prompt はサーバー側で固定
prompt = HARDCODED_PROMPT
```

重要な原則:
1. `webhook_id` を確認して通常ユーザーと区別する
2. webhook の本文で任意コマンドを指定させない
3. Claude へ渡す prompt はサーバー側で固定する

## データベースの安全性

- SQLite には主に `thread_id -> session_id` の対応だけを保存する
- メッセージ本文や秘密情報は原則保存しない
- SQL はすべて `?` プレースホルダを使う
- `cleanup_old()` で古いデータを削除できる

## デプロイ時の推奨事項

1. できればプライベートな Discord サーバーで運用する
2. Claude 用の専用チャンネルを分ける
3. 本番では `allowed_user_ids` を設定する
4. `permission_mode` や `allowed_tools` を見直す
5. `dangerously_skip_permissions` は理解した上でのみ使う
6. ログを監視する
7. 依存関係を定期的に更新する

## セキュリティ監査チェックリスト

`runner.py`、`_run_helper.py`、各 Cog に変更を入れる前後では、少なくとも以下を確認します。

- [ ] `shell=True` を使っていない
- [ ] ユーザー入力の前に `--` セパレータがある
- [ ] 外部入力 (session_id、skill 名、channel_id など) を検証している
- [ ] `_STRIPPED_ENV_KEYS` に新しい秘密情報を追加し忘れていない
- [ ] SQL を文字列連結で組み立てていない
- [ ] 新しいメッセージ入口に `allowed_user_ids` チェックがある
- [ ] `os.system()`、`subprocess.run(shell=True)`、`eval()` を導入していない
