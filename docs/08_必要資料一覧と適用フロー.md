# 必要資料一覧と適用フロー

## 目的

この文書は、ここまで整理した内容をもとに、個人開発の中小規模アプリケーションで一般的に必要となる資料を示し、それらをどの開発フローで、どの段階に作るかをまとめたものである。

対象は次のような開発である。

- API
- 中小規模 Web アプリケーション
- GUI を含む個人開発

この文書は、後続でスキル化するための前提資料として使う。

## 基本的な考え方

このプロジェクトでは、資料を次の 2 種類に分けて扱う。

### 1. 正式文書

`docs/` 配下に置く。人が読むための文書であり、あとで自分が見返して理解しやすいことを目的とする。

特徴:

- 固定ファイル名
- 日付で増やさない
- プロジェクトの説明責務を持つ
- セッションをまたいで読み継げる

### 2. 作業文書

`docs/superpowers/` 配下に置く。AI や開発セッションが作業を進めるための中間文書である。

特徴:

- テーマ単位で作る
- 日付付きでよい
- 設計、計画、進捗のために使う
- 最終的な説明責務は持たない

## 一般的に必要な資料

中小規模の個人開発で「過不足がない」ことを狙う場合、次の資料群が基本セットになる。

### A. 必ず考えるべき資料

以下は、毎回その要否を検討すべき資料である。

1. プロジェクト概要
2. 要件または設計の起点となる文書
3. アーキテクチャ
4. 設計判断
5. 利用ガイド
6. 実装計画
7. 進捗メモ

### B. 条件付きで必要になる資料

以下は、案件の性質によって必要性が高くなる資料である。

1. UI 設計
   - GUI がある場合
2. API 仕様
   - API を提供する場合
3. セキュリティと運用注意
   - 認証、外部 API、個人情報、スクレイピング、課金などがある場合
4. テスト計画
   - 回帰確認が複雑な場合
5. Runbook
   - バッチ、デプロイ、障害対応がある場合

## 推奨する正式文書

中小規模の個人開発に対して、過不足がない標準としては次を推奨する。

### 基本セット

- `docs/00_成果物戦略と開発フロー.md`
- `docs/01_プロジェクト概要.md`
- `docs/02_アーキテクチャ.md`
- `docs/03_設計判断.md`
- `docs/04_利用ガイド.md`

### 条件付きセット

- `docs/05_UI設計.md`
  - GUI がある場合
- `docs/06_API仕様.md`
  - API がある場合
- `docs/07_セキュリティと運用注意.md`
  - 注意事項が多い場合
- `docs/08_テスト方針.md`
  - テスト観点の共有が必要な場合
- `docs/09_Runbook.md`
  - 運用や障害対応がある場合

ファイル名は例であり、要点は「固定名の正式文書として育てる」ことである。

## 推奨する作業文書

テーマごとに、少なくとも次を作る。

- `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- `docs/superpowers/plans/YYYY-MM-DD-<topic>-implementation-plan.md`
- `docs/superpowers/YYYY-MM-DD-<topic>-progress-summary.md`

必要なら追加する。

- 画面別の設計メモ
- 調査メモ
- 検証メモ

## 既存フローへの当てはめ

ここでは、既存 Superpowers と、追加予定の上位スキルを合わせた流れに資料を当てはめる。

### フロー全体

1. `artifact-planning`
2. `brainstorming`
3. `using-git-worktrees`
4. `writing-plans`
5. `executing-plans` / `subagent-driven-development`
6. `test-driven-development`
7. `requesting-code-review`
8. `dual-track-documentation`
9. `verification-before-completion`
10. `finishing-a-development-branch`

## フェーズごとに何を作るか

### フェーズ 1. `artifact-planning`

目的:

- 何を作るかではなく、何を成果物として残すかを決める

この段階で決めるもの:

- 今回のテーマ
- 正式文書の更新対象
- 作業文書の作成対象
- GUI の有無
- API の有無
- セキュリティ文書の必要性
- テスト方針文書の必要性
- Runbook の必要性

この段階で作るもの:

- 初期の成果物一覧
- `progress-summary` の雛形

### フェーズ 2. `brainstorming`

目的:

- テーマの設計を固める

この段階で作るもの:

- 設計書

この段階で決めるもの:

- 採用アーキテクチャ
- 主要な設計判断
- GUI がある場合は画面構成の方向性
- API がある場合は公開インターフェースの方針

後で反映される正式文書:

- アーキテクチャ
- 設計判断
- UI 設計
- API 仕様

### フェーズ 3. `using-git-worktrees`

目的:

- 実装環境を隔離する

この段階で残すべき情報:

- ブランチ名
- worktree の有無
- baseline の検証結果

これは主に `progress-summary` に記録する。

### フェーズ 4. `writing-plans`

目的:

- 実装に落とす

この段階で作るもの:

- 実装計画
- タスク分解
- 検証手順

この段階で更新するもの:

- `progress-summary`

### フェーズ 5. `executing-plans` / `test-driven-development`

目的:

- コードとテストを進める

この段階で更新するもの:

- コード
- テスト
- `progress-summary`

この段階で後で反映される正式文書:

- 利用ガイド
- API 仕様
- UI 設計
- セキュリティと運用注意

### フェーズ 6. `requesting-code-review`

目的:

- 実装の妥当性と不足を洗う

この段階で残すべき内容:

- 指摘事項
- 未解決事項
- 次の修正対象

これは `progress-summary` またはレビュー記録に残す。

### フェーズ 7. `dual-track-documentation`

目的:

- 作業文書の知識を正式文書へ反映する

この段階で更新するもの:

- アーキテクチャ
- 設計判断
- 利用ガイド
- UI 設計
- API 仕様
- セキュリティと運用注意

この段階で確認すること:

- `docs/superpowers` に閉じた知識が残っていないか
- 次回再開時に `docs/` 側だけでも全体像が追えるか

### フェーズ 8. `verification-before-completion`

目的:

- 完了主張に根拠を持たせる

この段階で残すもの:

- 実行した確認コマンド
- 通過したテスト
- 未確認項目

### フェーズ 9. `finishing-a-development-branch`

目的:

- 最終 handoff と完了処理

この段階で残すもの:

- 最終の `progress-summary`
- 文書未反映がある場合はその一覧
- マージまたは PR 判断

## API 案件で必要になりやすい資料

API 案件では次が特に重要である。

- アーキテクチャ
- 設計判断
- API 仕様
- 利用ガイド
- セキュリティと運用注意
- 設計書
- 実装計画
- 進捗メモ

今回の `jra-srb` はこの型に近い。

## GUI 案件で必要になりやすい資料

GUI を含む Web アプリでは次が特に重要である。

- プロジェクト概要
- アーキテクチャ
- 設計判断
- UI 設計
- 利用ガイド
- API 仕様
- セキュリティと運用注意
- 設計書
- 実装計画
- 進捗メモ

必要に応じて追加:

- テスト方針
- Runbook

## 何を作るかを判断する基準

資料を作るか迷ったときは、次を基準に判断する。

1. 後で見返したときに思い出せない可能性が高いか
2. 実装前に整理しておかないと迷いやすいか
3. 影響範囲が広いか
4. 別セッション再開時に詰まりやすいか
5. コードだけでは読み取りにくいか

これらに多く当てはまるものは、正式文書または作業文書として残す価値が高い。

## 過不足がない標準

このプロジェクトで目指すのは「最小構成」ではなく「過不足がない構成」である。

その意味は次のとおりである。

- 毎回同じ資料一式を機械的に作らない
- ただし、後で自分が見て分からなくなるものは残す
- 作業文書だけで閉じない
- 正式文書だけで現場の進捗を管理しようとしない

したがって、標準は「固定の全資料作成」ではなく、

- 最初に必要資料を決める
- 作業の中で更新する
- 節目で正式文書へ反映する

という運用になる。

## 結論

個人開発の中小規模アプリケーションであっても、最初に「何を成果物として作るか」を決めるフェーズはあった方がよい。

その上で、必要資料は次の 3 層で考えるのがよい。

1. 正式文書
2. 作業文書
3. セッション継続のための進捗情報

そして、既存の実装フローに加えて、最初に成果物計画を置き、途中で `update-docs` 的に正式文書へ反映する流れを入れることで、複数セッション前提でも破綻しにくい運用になる。
